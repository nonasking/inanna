# Inanna — 요건 정의

> 2026-07-10 초안. 선행 조사: [research.md](research.md)

## 1. 비전

**캐릭터가 아니라 관계를 만든다.**

기존 서비스(Character.AI, 제타, Cotomo, Grok Ani)는 전부 '캐릭터'를 만들고 사용자가 그 캐릭터를 소비한다. Inanna는 반대로 **관계 유형(연인/친구/동생/…)을 1급 개념**으로 두고, 성격·말투·호칭·거리감이 관계에서 파생되게 한다. 그 위에 사용자가 성격·모습·목소리를 자유롭게 커스텀한다.

세 가지 축을 하나로:
1. **관계 & 페르소나 커스텀** — 관계 유형 + 성격/성향/말투를 사용자가 정의
2. **목소리** — 참조 오디오 기반 보이스 클로닝 또는 프리셋으로 캐릭터 음성 대화
3. **소유** — 셀프호스팅. 관계 데이터(기억)는 사용자 것. 서비스가 죽어도 관계는 죽지 않는다 (Gatebox 미쿠의 반대)

**모습은 의도적으로 없다.** 목소리+텍스트만으로 존재하는 무형 컴패니언 — 모습은 사용자의 상상에 맡긴다(2026-07-10 확정). 라디오 드라마 원리: 어중간한 비주얼은 상상을 끌어내리지만, 좋은 목소리는 상상을 밀어올린다. 부수 효과로 저작권상 침해 판단이 가장 쉬운 축(외형)이 제거되고, 아바타 파이프라인 스코프가 통째로 사라져 관계 엔진·음성 품질에 집중할 수 있다.

## 1.5 형태와 배포 전략 (2026-07-10 확정: App Store 배포를 염두)

**최종형 = iOS 네이티브 앱 (App Store 배포).** 단, 투트랙으로 간다:

| 트랙 | 형태 | 용도 |
|---|---|---|
| **개발 모드** (현재) | 맥 셀프호스팅 서버 + 웹 클라이언트 | P0~P2 기능 검증. 수정-새로고침 속도 우선 |
| **제품 모드** (P4) | 호스티드 백엔드 + SwiftUI 네이티브 앱 | App Store 배포. 계정·푸시(선제 발화)·백그라운드 음성 |

설계 원칙: **모든 로직과 상태는 서버, 클라이언트는 얇은 뷰.** 웹 클라이언트는 갈아끼울 수 있어야 하고, API가 곧 계약이다. 이를 위해 P0부터 선반영하는 것: ① 모든 데이터를 user_id로 스코핑(셀프호스팅은 단일 유저 `local`), ② API bearer 인증(제품 모드에서 계정 인증으로 교체), ③ P2 음성 인터페이스는 네이티브에서 그대로 쓸 수 있는 WebSocket 오디오 스트림으로 설계.

### App Store 제약 체크리스트 (제품 모드 진입 전 필수)
- [ ] **NSFW 기능화 금지** — 이미 비목표(§6). App Store에서는 협상 불가 조건
- [ ] **연령 등급 17+** + 연령 게이트 — AI 컴패니언 앱 관행 + 미성년 안전 소송 선례(Character.AI Garcia 사건) 대응
- [ ] **계정 삭제 기능** — 계정 생성이 있으면 앱 내 삭제 필수(가이드라인 5.1.1(v)). 데이터 소유 원칙과 일치: 내보내기+완전 삭제
- [ ] **Sign in with Apple** — 서드파티 소셜 로그인 제공 시 의무
- [ ] **개인정보 처리방침 + 대화 데이터 보관 정책** — 대화는 민감 데이터. 저장 위치·암호화·보관 기간 명시
- [ ] **저작권(호스티드 전환 시)**: 사용자 콘텐츠가 서버에 저장되는 순간 플랫폼 책임 발생. 1차 방어 = **공유 기능 없음 유지**(각자의 컴패니언은 본인만 접근 — 타인 노출 UGC가 없으면 notice-and-takedown 부담이 최소화). 공유 도입은 신고/모더레이션 체계와 세트로만

## 2. 저작권 포지션 (확정)

- **플랫폼(프로젝트)은 어떤 IP도 배포하지 않는다.** 기본 제공 캐릭터는 오리지널만.
- 성격·목소리는 **자유 커스텀으로 완전히 열어둔다.** 사용자가 무엇을 입히는지는 사용자 영역(로컬/셀프호스팅이므로 SillyTavern과 같은 위치 — 사실상 안전지대). 외형은 아예 다루지 않으므로(§1) 저작권상 가장 민감한 축이 설계 단계에서 제거됨.
- 커뮤니티 공유/카드 배포 기능은 **후순위**로 미룬다. 도입 시 notice-and-takedown 체계가 선행돼야 함 (디즈니→Character.AI, 웹툰6사→제타 선례).
- 실존 인물(성우 포함) 목소리 클로닝을 기능으로 **홍보하지 않는다** — 퍼블리시티권 리스크. 기능 자체는 "참조 오디오로 목소리 커스텀"으로 중립 제공.

## 3. 핵심 개념 모델

```
Companion(동반자)
├── Relationship  관계 유형(연인/친구/동생/누나/선배/집사/…) ← 1급 개념
│     ├── 호칭 (나를 뭐라 부르는가 / 내가 뭐라 부르는가)
│     ├── 말단계 (반말/존댓말/혼합)
│     ├── 거리감 (스킨십·애정표현·장난 허용 수위)
│     └── 관계 서사 (어떻게 만난 사이인가)
├── Persona       성격·성향·말투
│     ├── 트레이트 (밝음↔차분, 직설↔돌려말함, 애교, 유머, …)
│     ├── 말버릇/어미 (예: "~다니까?", 이모지 습관)
│     └── 배경 설정 + 로어북 (키워드 트리거 지식)
├── Voice         목소리 (모습이 없으므로 = 컴패니언의 유일한 감각적 정체성)
│     ├── 프리셋 보이스 or 참조 오디오(클로닝)
│     └── 파라미터 (속도, 톤)
└── Memory        관계의 축적
      ├── Core: 페르소나+관계 요약 (항상 주입)
      ├── Episodic: 대화 요약·사건 (최근성 가중)
      └── Archival: 검색형 장기 기억 (임베딩 recall)
```

관계 유형은 **프리셋 템플릿**(연인/친구/동생 등)으로 시작하되, 모든 파생값을 사용자가 오버라이드 가능. 템플릿은 출발점이지 제약이 아니다.

## 4. 기능 요구사항

### P0 — 관계 엔진 + 텍스트 대화 (MVP)
- [ ] Companion 정의 스키마 (YAML/JSON): relationship / persona / voice / memory 섹션
- [ ] 관계 템플릿 5종: 연인, 친구, 여동생/남동생, 누나/형, 비서
- [ ] 프롬프트 컴파일러: 스키마 → 시스템 프롬프트 (호칭·말단계·거리감이 실제로 지켜지는지가 품질 기준)
- [ ] 대화 루프: 웹 채팅 UI (모바일 대응 — 폰에서 Tailscale로 접속)
- [ ] **미리보기 대화**: 빌더에서 저장 전 현재 설정으로 몇 턴 시험 대화 → 조정 → 재시험 루프 (기억에 남지 않는 임시 대화). 커스텀 경험의 핵심 반복
- [ ] 메모리 v0: 세션 대화 이력 + 주기적 요약을 다음 세션에 주입
- [ ] LLM 프로바이더 추상화: Anthropic API 기본, Ollama(로컬) 옵션
- [ ] CCv2/v3 캐릭터 카드 **임포트** (기존 생태계 흡수 — persona 필드 매핑)

### 품질 검증 기록 (2026-07-10, 자동 테스트)
- ✅ 기준 #1 관계 파생 (P0에서 검증) · #2 10턴 말투 유지 (존댓말 누출 0/10, 호칭 10/10) · #3 기억 이월 (요약→새 세션 주입, 약속·디테일 정확 회상)
- ⚠ 관찰: 없는 기억을 지어내는 가벼운 컨패뷸레이션 ("지난번에 먹고싶다던 족발") — 컴패니언답게 자연스럽지만, 기억에 없는 과거는 단정하지 말라는 프롬프트 가드 검토 (P2)
- P2 스택 사전 검증: whisper.cpp(small, Metal)가 클로닝 음성 4초를 1.2초에 한국어 인식(거의 무손실). STT 1.2s + TTS RTF<1 → 실시간 루프 실현 가능

### P2 — 실시간 음성 대화 + 기억 고도화
- [x] **음성 통화 루프 구현 (2026-07-10)**: WS 프로토콜(docs/voice-protocol.md, P4 계약) — 반이중 핸즈프리(사용자 확정), EnergyVAD, whisper-server 상주(:9881), 기존 chat_stream 재사용(음성 턴도 기억 대상), 문장 단위 TTS 스트리밍, 탭 인터럽트(부분 응답 저장). e2e 5개 시나리오 통과, 발화종료→첫오디오 ~3.5s. HTTPS: tailscale serve (iOS 마이크 필수)
- [ ] 폰 실기기 검증 (마이크 권한·오디오 언락·wake lock) — **사용자 테스트 대기**
- [ ] barge-in 승격 검토 (프로토콜 변경 없음 — speaking 중 수신 허용 + 서버 VAD 자동 interrupt)
- [x] **관련 기억 recall (2026-07-10)**: 최근 5개 + BM25(한글 바이그램) 관련 5개 병합 주입 (`server/memory/recall.py`). 임베딩은 Python 3.14 venv에서 torch/onnx 휠 부재로 보류 — get_relevant만 교체하면 되는 구조. e2e: 최근 밖의 '부산 여행' 기억을 질문으로 소환 성공
- [x] **관계 진행 (2026-07-10)**: 함께한 일수·기념일(50/100/…일)·오랜만(3일+) 감지·현재 시각/요일을 '지금 상황' 블록으로 주입 (`server/chat/relationship.py`). 세션 요약에 '분위기' 톤 라인 추가. 사용자 설정(intimacy)은 불변 — 시간의 흐름만 상황으로 전달
- [x] **컨패뷸레이션 가드 (2026-07-10)**: 기억/대화에 없는 과거 단정 금지 대화 원칙. e2e: 없는 크리스마스 기억을 물으면 지어내지 않고 되물음. 관찰: 실재 기억 위에 디테일을 얹는 각색은 여전히 있음(대화체로는 자연스러워 v1 허용)

## 5. 비기능 요구사항

- **셀프호스팅 1순위**: 맥미니 한 대에서 전부 구동 가능해야 함. 외부 의존은 LLM API 정도만 허용(그마저 Ollama로 대체 가능하게).
- **모바일 접근**: 기존 Tailscale 환경으로 폰에서 사용 (별도 앱 없이 PWA/웹).
- **데이터 소유**: 모든 기억·설정은 로컬 파일/DB(SQLite). 내보내기 가능.
- **비용 0원 지향**: 유료 API는 옵션이지 필수가 아니어야 함.
- **라이선스**: Apache-2.0 (오리지널 프로젝트 컨벤션). SillyTavern 코드는 AGPL이므로 **코드 재사용 금지, 개념만 참고**. Fish Speech 가중치(CC-BY-NC) 상용 불가 유의.

## 6. 명시적 비목표 (당분간)

- **아바타/외형 일체** (이미지·Live2D·VRM) — §1의 설계 원칙. 원하는 사용자가 생기면 외부 클라이언트가 붙일 수 있는 정도의 여지만 남기고, 코어는 영원히 무형
- 캐릭터 카드 마켓플레이스/공유 커뮤니티 (저작권 대응 체계 없이는 안 함)
- 멀티유저 SaaS
- NSFW 콘텐츠 기능화 (Grok Ani의 규제 궤적 참고)
- 실존 인물 음성 프리셋 제공
